## Executive Summary

**Problem:** Voice PTT (Push-To-Talk) did not stop recording when the user released the button. The recording would continue indefinitely until the 300-second subprocess timeout or app restart.

**Root cause:** A race condition in `ui-v8/app/main_window.py` `_handle_voice_ptt_released()`. The function checked `self._latest_state_snapshot.voice_runtime.get("listening", False)` before sending a stop command. Since the snapshot is only updated by a 1-second polling timer (`refresh_runtime_state()`), a quick press-and-release could release the button **before** the next poll updated the snapshot. The release was then silently ignored, and the backend continued recording forever (no max duration, no VAD).

**Fix:** Replaced the snapshot-dependent check with a local `_voice_ptt_active` flag that tracks PTT state directly in the UI. Also added 60-second max recording duration safety caps to both recording backends.

## 1. UI Race Condition Fix ✅

### File: `ui-v8/app/main_window.py`

**Before (buggy):**
```python
def _handle_voice_ptt_released(self) -> None:
    ...
    if not bool(self._latest_state_snapshot.voice_runtime.get("listening", False)):
        return   # ← silently ignored if snapshot stale!
    ...
```

**After (fixed):**
```python
def _handle_voice_ptt_released(self) -> None:
    ...
    if not self._voice_ptt_active:
        return   # ← uses local flag, never stale
    ...
```

### `_voice_ptt_active` state machine

| Event | Action | `_voice_ptt_active` |
|-------|--------|---------------------|
| PTT pressed, start initiated | Set flag | `True` |
| PTT released | Check flag, send stop | `True` → stays True until stop completes |
| Start fails / timeouts | Clear flag | `False` |
| Stop completes (ok/timeout/fail) | Clear flag | `False` |
| Start succeeds + queued release | Auto-trigger stop | `True` (stop handler clears) |

### Additional robustness improvements

1. **Start guard updated** (line 499): Also uses `_voice_ptt_active` to prevent double-starting, instead of relying on the stale snapshot.
2. **Auto-trigger path updated** (line 649): Uses `_voice_ptt_active` instead of snapshot `listening` flag when deciding whether to auto-trigger a queued release after start completes.

## 2. Runaway Recording Safety Caps ✅

Even with the race condition fixed, a lost stop signal (e.g., subprocess crash, signal not delivered) would cause infinite recording. Two safety caps were added:

### `tools/router_py/voice_recorder.py` (Python voice path)

```python
MAX_RECORDING_DURATION_SECONDS = 60

def record_audio(..., max_duration_seconds: Optional[int] = None):
    max_dur = max_duration_seconds if max_duration_seconds is not None else MAX_RECORDING_DURATION_SECONDS
    recording_start_time = time.monotonic()
    while _recording and not check_stop_signal(stop_file):
        if time.monotonic() - recording_start_time >= max_dur:
            logger.warning(f"Recording stopped after reaching max duration ({max_dur}s)")
            break
        ...
```

### `tools/runtime_voice.py` `run_internal_recorder()` (Shell voice path)

```python
def run_internal_recorder(output_path: Path, runtime_file: Path, max_duration_seconds: int = 60) -> int:
    recording_start_time = time.monotonic()
    while not _INTERNAL_RECORD_STOP:
        if time.monotonic() - recording_start_time >= max_duration_seconds:
            print(f"internal recorder stopped after reaching max duration ({max_duration_seconds}s)", file=sys.stderr)
            break
        ...
```

**Note:** The Python-native `StreamingVoicePipeline` in `voice_runtime.py` already had an 8-second cap via `LUCY_VOICE_MAX_SECONDS` (default). Only the standalone recorder paths needed the safety cap.
