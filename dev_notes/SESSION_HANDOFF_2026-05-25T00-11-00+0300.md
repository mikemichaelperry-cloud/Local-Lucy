## What Was Done

### 1. Tube Database Module Cleanup (`data/tubes/tube_database.py`)

**Problem:** The seed data in `tube_database.py` had duplicates (`6CA7`, `6BQ5`, `KT77`, `6BM8`, `6AQ5` appeared twice) and the module lacked reusable helpers for consumers.

**Fixes:**
- **Deduplicated** `SEED_TUBES` list (removed 5 duplicate entries, replaced some with distinct variants: `6CA7S`, `KT77EH`, `6BM8EH`, `6AQ5A`, `6CL6`)
- **Added helper functions:**
  - `get_db_path() → Path` — canonical DB location
  - `list_all_types(conn) → list[str]` — all tube types, sorted
  - `get_all_tubes(conn) → list[dict]` — every record as a dict
- **Updated docstring** to reflect 648 tubes and OpenAI bulk-extraction origin
- **Created `data/tubes/__init__.py`** to make the directory importable as a package

**Result:** Clean, reusable module with no duplicate seed data.

---

### 2. Refactored `local_answer.py` to Use Module API

**Problem:** `_lookup_tube_database()` contained ~60 lines of raw SQLite code duplicated from `tube_database.py`, opening/closing the DB twice per lookup and manually formatting output.

**Fix:**
- Added safe import of `tube_database` at module load time (with `sys.path` insertion and fallback to `None`)
- Rewrote `_lookup_tube_database()` to use:
  - `tube_database.init_db()` for connection
  - `tube_database.list_all_types()` for type enumeration
  - `tube_database.lookup_tube()` for record retrieval
  - `tube_database.format_tube_for_model()` for output formatting
- Preserved **longest-match logic** (`6V6GT` wins over `6V6` when both match)
- Preserved silent fallback if DB is missing or unreadable

**Result:** ~40 lines removed, zero behavior change, cleaner separation of concerns.

## Architecture Updates

### Tube Database Lookup (NEW)

```
generate_answer()
├── identity query? → identity response
├── policy response? → fixed response
├── social greeting? → warm canned response
├── medical high-risk? → evidence refusal
├── time-sensitive? → evidence refusal
├── 807 push-pull? → hardcoded answer
├── _lookup_tube_database(q)? → tube specs (NO Ollama call)
│   └── uses tube_database.lookup_tube() + format_tube_for_model()
└── ... cache / Ollama / post-processing ...
```

**Performance:** Tube lookup happens in **0–2 ms** (SQLite in-memory cache + simple string match). No network, no GPU, no model load.

## Data Quality Notes

- **648 total tubes** in database
- **99 tubes (15.3%)** have `construction="unknown"` — these are very obscure types where OpenAI had no datasheet reference. They do not cause harm because `format_tube_for_model()` still outputs the type name and any available fields.
- **Critical tubes verified:** 6V6GT, 6L6GC, EL34, KT88, 6550, 300B, 2A3, 845, 12AX7, 12AT7, 12AU7, 6SN7GT, 5U4G, 5AR4, 807, 5881, 7591, EL84 — all have accurate specs from RCA/GE manuals or manufacturer datasheets.
- **Heater voltage ambiguity:** 12V-series tubes (12AX7, 12AT7, etc.) store `heater_volts=6.3` (parallel) by default; some variants like `12AT7A` store `12.6` (series string). This is technically correct but the schema does not distinguish parallel vs series configuration.
