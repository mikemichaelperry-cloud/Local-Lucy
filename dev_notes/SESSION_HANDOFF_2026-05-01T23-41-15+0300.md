### Initial Result: 33 passed, 1 failed

**Failure:** Rapid model switch — switches 1, 3, 5 (all qwen3) returned "Empty response from Ollama".

### Root Cause Analysis

The failure was **NOT** Ollama model loading time or concurrency races. It was **Qwen3's thinking mode consuming the token budget**:

1. Thrash test query: `"Say hello briefly."`
2. `"briefly"` matched `brief_patterns` in `_set_generation_profile()` → `num_predict_brief = 64` tokens
3. Qwen3 14B emits reasoning tokens in a separate `thinking` field before generating `response`
4. With only 64 tokens, Qwen3 filled the entire budget with thinking tokens → `response` empty
5. Ollama returned: `{"response": "", "thinking": "Okay, the user said...", "done_reason": "length", "eval_count": 64}`
6. All retry attempts also used 64 tokens, so they all failed identically
7. The 8B model (llama3.1) doesn't have this thinking behavior, so it worked fine with 64 tokens

### Fix Applied

**`tools/router_py/local_answer.py`**:
- `num_predict_brief` increased `48 → 128` (default and `from_env()`)
- Added `thinking` fallback: if `response` is empty but `thinking` is present, use the thinking text instead of failing completely
- Added module-level `threading.Lock()` around `_call_ollama`
- Added 5-attempt exponential backoff retry (2s, 4s, 8s, 16s) for genuine model load/unload transitions

**Result after fix:** `34 passed, 0 failed` ✅

## 2. Empty Query Guard Fix ✅

### Problem

The empty query guard was placed **after** the `use_python_path` async branch in `execute()`. When `use_python_path=True` (the default UI path), the guard was bypassed entirely. Empty or whitespace-only queries returned `status="completed"` with fallback text instead of `status="failed"` with `outcome_code="empty_query"`.

### Fix

**`tools/router_py/execution_engine.py`**:
- Moved empty query guard to the **very top** of `execute()`, before any route dispatch
- Added identical guard to `execute_async()` for defense in depth

```python
# Reject empty or whitespace-only queries at the engine boundary
if not question or not question.strip():
    return ExecutionResult(
        status="failed",
        outcome_code="empty_query",
        ...
    )
```

**Verified:** Both sync (`use_python_path=False`) and async (`use_python_path=True`) paths now correctly return `outcome_code="empty_query"`.

## 3. Critical Bug: Symlinked runtime_voice.py ✅

### Problem (Would Have Broken Live Testing Tomorrow)

`snapshots/opt-experimental-v9-dev/tools/runtime_voice.py` was a **symlink** pointing to `lucy-v8/tools/runtime_voice.py`.

When the UI called the snapshot script via subprocess:
1. Python resolved the symlink
2. `sys.path[0]` became `lucy-v8/tools/` (workspace) instead of `snapshots/.../tools/` (snapshot)
3. `from runtime_control import enforce_authority_contract` loaded the **workspace** version
4. The workspace `runtime_control.py` had `__file__` resolving to `lucy-v8/`
5. This **mismatched** `LUCY_RUNTIME_AUTHORITY_ROOT=snapshots/opt-experimental-v9-dev/`
6. **Every voice PTT start/stop and TTS speak command failed with:**
   ```
   RuntimeControlError: authority root mismatch: env=/home/mike/.../snapshots/opt-experimental-v9-dev expected=/home/mike/lucy-v9
   ```

### Fix

- Deleted the symlink
- Copied the real `tools/runtime_voice.py` into the snapshot
- Verified: `python3 snapshot/tools/runtime_voice.py status` now passes authority contract
- Verified: `python3 snapshot/tools/runtime_control.py set-model` now passes authority contract

**Note:** `snapshots/.../tools/voice` is still a symlink to `lucy-v8/tools/voice`, but this is a **directory** of shared voice utilities (`playback.py`, `tts_adapter.py`, etc.) and the package imports handle this correctly.
