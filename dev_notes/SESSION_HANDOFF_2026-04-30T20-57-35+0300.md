## 1. Voice PTT Stop Fix ✅

### Root cause

`_handle_voice_ptt_released()` checked `self._latest_state_snapshot.voice_runtime.get("listening", False)` before sending a stop command. The snapshot is only updated by a **1-second polling timer** (`refresh_runtime_state()`). A quick press-and-release (< 1s) could release the button **before** the next poll updated the snapshot. The release was silently ignored, and the backend recorded forever (no max duration, no VAD).

### Fix: `_voice_ptt_active` local flag

Replaced the snapshot-dependent check with a local `_voice_ptt_active` boolean that tracks PTT intent directly in the UI:

| Event | `_voice_ptt_active` |
|-------|---------------------|
| PTT pressed, start initiated | `True` |
| PTT released | Checked (not snapshot) |
| Start fails / timeouts | `False` |
| Stop completes | `False` |

**Files modified:**
```
ui-v8/app/main_window.py  # +_voice_ptt_active flag, snapshot-free guards, timer hooks
```

### 60s safety cap added to both recorder paths

Even with the race fixed, a lost stop signal (subprocess crash, signal not delivered) would cause infinite recording:

```
tools/router_py/voice_recorder.py     # MAX_RECORDING_DURATION_SECONDS = 60 (now env-configurable)
tools/runtime_voice.py                # max_duration_seconds param on run_internal_recorder()
```

## 5. Performance Baseline (Parallel Exploration) 🔍

Three explore agents mapped the full system. Key findings:

### 5.1 Model Configuration

| Parameter | Current Value |
|-----------|---------------|
| Active model | `local-lucy` (custom Ollama model based on `llama3.1:8b`) |
| Context window | 2048 tokens |
| Temperature | 0.5 |
| Top-p | 0.5 |
| Max output tokens (chat) | 1536 |
| GPU | RTX 3060 12 GB |
| Free VRAM (idle) | ~5.2 GB |

### 5.2 Critical Performance Bottlenecks

**1. Response cache is effectively useless**
- 5-entry limit, 60-second TTL, disk-based file I/O
- Guard logs show "Ohm's law" repeated 80+ times — cache evicts it within a minute
- **Quick win:** Increase to 100+ entries, 300-900s TTL, or move to in-memory LRU

**2. Every augmented request spawns a subprocess**
- `_call_single_provider()` spawns Python interpreter + tool script
- 130s timeout means failures hang for minutes
- **Quick win:** Replace with direct `aiohttp` async calls

**3. Wikipedia evidence uses sync urllib in thread pool**
- Up to 3 sequential HTTP requests × 2.5s = 7.5s worst case
- **Quick win:** Replace with `aiohttp`

**4. File-locked dual state writes on every request**
- `fcntl.flock` with exponential backoff (10ms → 50ms → 100ms)
- SQLite write + `.env` file write on every request
- **Quick win:** Skip `.env` writes when `use_sqlite_state=True`

### 5.3 Voice Pipeline Optimization Opportunities (RTX 3060)

| Opportunity | Current | Recommendation | Effort | Impact |
|-------------|---------|----------------|--------|--------|
| Whisper model | `small.en` (488 MB) | **`large-v3-turbo`** (1.62 GB, already downloaded) | Trivial | High — better accuracy |
| Whisper beam size | `1` (greedy) | `LUCY_VOICE_WHISPER_BEAM_SIZE=5` | Env var | Medium — better accuracy |
| Whisper threads | `4` | `8` (in `whisper_worker.py`) | 1 line | Low-Medium |
| Kokoro device | Auto-detect | `LUCY_VOICE_KOKORO_DEVICE=cuda` | Env var | Low — prevents CPU fallback |
| Piper | CPU only | ONNX Runtime GPU (advanced) | Medium | Medium — frees CPU |
| VAD in PTT | Not wired | Wire `_record_with_vad()` as secondary stop | Medium | High — auto-stop on silence |
| Pre-warm workers | At PTT-start | At app boot | Medium | Low — eliminates first-latency |

**Bottom line:** The voice pipeline is well-structured with automatic GPU fallback, but it runs a **small.en** Whisper model with **greedy decoding (beam=1)** and **4 threads** — very conservative for an RTX 3060. The biggest wins are switching to the already-present **`large-v3-turbo`** model and/or increasing beam size.
