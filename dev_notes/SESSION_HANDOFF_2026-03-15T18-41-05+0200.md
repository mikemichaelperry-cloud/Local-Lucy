## Root Cause

### Medical matcher duplication leak

The router-side medical matcher in `tools/router/core/intent_classifier.py` had drifted from the execution defense-in-depth matcher in `tools/query_policy.sh`. That allowed:

- dry route: `PIPELINE=LOCAL`
- later execution refusal: `This requires evidence mode...`

for prompts such as:

- `What are the side effects of ibuprofen?`

### Semantic provenance issue

`tools/router/plan_to_pipeline.py` could pass a rewritten contextual followup form into semantic trace output as `original_query`, losing literal user prompt provenance.

## Exact Fix Shape

### Matcher reuse

The router classifier now reuses shared human-medication high-risk matching logic rather than maintaining a narrower duplicate list. `query_policy.sh` also reuses the same shared logic through the thin Python wrapper, while keeping pet/veterinary checks local. This keeps governor/router as the primary owner of the contract boundary and leaves execution checks as defense-in-depth only.

### Provenance

Semantic trace now carries both:

- `original_query`
- `resolved_execution_query`

`original_query` is always set from the literal incoming user prompt. The resolved form is exposed separately for contextual followup rewriting.
