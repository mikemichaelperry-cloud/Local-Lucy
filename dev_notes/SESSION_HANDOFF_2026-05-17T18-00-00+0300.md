## What Was Done in This Session

### 1. Response Regression Test Suite

**Problem:** After Modelfile/Ollama config changes, there was no automated way to verify Lucy still answered key questions correctly (identity, capabilities, personality, religious neutrality).

**Solution:**
- Created `tests/response_regression_cases.json` — 10 test cases with structural checks
- Created `tools/router_py/test_response_regression.py` — pytest + direct CLI runner
- Records "golden" responses with `LUCY_RESPONSE_REGRESSION_RECORD=1`
- Compares new responses against baseline; fails on structural check violations or text diffs
- Checks: `must_include_regexes`, `must_exclude_regexes`, `first_person_only`, `max_chars`
- Golden responses stored in `tests/golden_responses.json`

**Known failures (3):** `personality_logic_over_ideology`, `reasoning_structured`, `memory_respect` — these document that first-person mandate is still inconsistent in some query paths. Fix the Modelfile, re-record.

### 2. HMI Enter Key Submit

**Problem:** The Submit button in the conversation panel required mouse click; no keyboard shortcut.

**Solution:**
- Added `eventFilter` on `self._draft` (QTextEdit) in `conversation_panel.py`
- `Enter` / `Return` triggers `_emit_submit_requested()`
- `Shift+Enter` passes through for newline insertion
- Updated placeholder text to hint the behavior

### 3. Model Toggle: `local-lucy-fast`

**Problem:** Selecting `local-lucy-fast` in HMI runtime toggles failed with: `invalid value 'local-lucy-fast' for model_selection`

**Root cause:** `runtime_bridge.py` whitelisted only `("local-lucy",)`, while UI dropdown and CLI tool both knew about `local-lucy-fast`.

**Fix:** Widened `allowed_values` to `("local-lucy", "local-lucy-fast")`.

### 4. `local-lucy-fast` Modelfile (Tracked)

**Problem:** `local-lucy-fast` was an untracked Ollama artifact with pre-May-16 personality (old "assistant" framing, no core values).

**Fix:**
- Created `config/Modelfile.local-lucy-fast` — identical SYSTEM prompt to `local-lucy`, but `num_ctx 1024`
- Updated `README.md` setup instructions
- Rebuilt model: `ollama create local-lucy-fast -f config/Modelfile.local-lucy-fast`

### 5. Model Persistence Across Restarts

**Problem:** `START_LUCY.sh` hardcoded `LUCY_LOCAL_MODEL=local-lucy-fast`. After toggling to `local-lucy` in HMI and restarting, launcher warned: `current_state.json model='local-lucy' differs from LUCY_LOCAL_MODEL='local-lucy-fast'`

**Fix:** `START_LUCY.sh` now reads model from `current_state.json` at startup, falling back to `local-lucy-fast` only if no state file exists.

### 6. Kimi Configuration Fix

**Problem:** No `.env` file existed; API keys only in interactive shell env. Desktop shortcut launches didn't see them. Also, `ui-v8/tools/unverified_context_kimi.py` lacked `MOONSHOT_API_KEY` fallback.

**Fix:**
- Created `.env` from existing shell environment (`OPENAI_API_KEY`, `MOONSHOT_API_KEY`)
- Fixed `ui-v8/tools/unverified_context_kimi.py` to check `MOONSHOT_API_KEY` (matching root copy)
- `.env` is `chmod 600`

### 7. AUGMENTED Fallback Chain

**Problem:** `_fetch_evidence` tried exactly **one** provider, then gave up and called local model with empty context. The "Wikipedia → OpenAI → Kimi fallback chain" existed only in `_call_augmented_provider`, which is used by a different code path.

**Fix:** Wired fallback chain into `_fetch_evidence` in `execution_engine.py`:
- Wikipedia → OpenAI → Kimi (general queries)
- Trusted → Wikipedia → OpenAI → Kimi (medical/vet)
- Route's preferred provider first, then iterates chain on failure

### 8. Fallback Chain Order Correction

**Problem:** Docs promised `wikipedia → openai → kimi`, but code did `wikipedia → kimi → openai`.

**Fix:** Corrected order in both `_fetch_evidence` and `_call_augmented_provider`.

### 9. Financial Keyword False Positive

**Problem:** `"ira"` (Individual Retirement Account) matched as substring inside `"iran"`, forcing `evidence_mode="required"` with `evidence_reason="financial_data"`. This overrode NEWS detection for Israel-Iran geopolitical queries.

**Fix:** Word-boundary matching for short (≤4 char) keywords in both `policy.py` and `hybrid_router.py`.

### 10. Geopolitical NEWS Routing

**Problem:** Israel-Iran military query routed to LOCAL instead of NEWS. Three causes:
- `"ira"` false positive (fixed above)
- `news_keywords` lacked conflict/geopolitics terms
- `embedding_local_override` pushed NEWS queries to AUGMENTED instead of NEWS

**Fix:**
- Added conflict terms to `news_keywords`: `military action`, `war`, `conflict`, `ceasefire`, `hostilities`, `invasion`, `airstrike`, `missile`, `rocket attack`, `bombing`, `shelling`, `troops`, `sanctions`, `embargo`, `diplomatic`, `tensions`, `escalation`, `retaliation`, `strike`
- Added `elif "NEWS" in candidate_routes` to `embedding_local_override` branch — routes to NEWS instead of AUGMENTED when intent classifier detected news
- Added `probability|likelihood|chance|odds|risk` to conflict-news prediction keywords in `intent_classifier.py`

### 11. Temporal Injection for API Providers

**Problem:** OpenAI/Kimi prompts asked for "background summaries" with no date context. Models answered from training cutoff, not current time. Result: generic "I don't have real-time data" boilerplate.

**Fix:**
- **System prompt** reframed: "You are a current-events analyst... Answer as of the current date."
- **Date injection:** `Today is 2026-05-17 18:00 UTC.` prepended to every user message
- **Augmented prompt** (`build_augmented_prompt`): Injects `Current date and time: ...` and instruction to use own knowledge if context is outdated
- Applied to both `unverified_context_openai.py` and `unverified_context_kimi.py`

### 12. API Provider Live Test Suite

**Problem:** No automated validation that OpenAI/Kimi actually return substantive, temporally-aware answers.

**Solution:**
- Created `tools/router_py/test_api_providers.py` — 15 tests
- 9 fast unit tests (no API calls)
- 6 live API tests (3 OpenAI, 3 Kimi) — run with `LUCY_TEST_LIVE_APIS=1`
- Validates: non-empty answers, temporal markers, Israel-Iran entity presence
